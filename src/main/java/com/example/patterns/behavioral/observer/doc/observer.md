# 观察者模式 Observer

业务场景：订单状态变更通知。一笔订单的状态发生变更（如已支付、已完成）后，需要触发一系列彼此独立的后续动作：向买家发送短信通知、为买家发放积分等。这些动作各自关注不同的业务、互不依赖，且未来可能随业务增减。

## 一、解决的问题

订单状态变更后往往要触发多个后续动作。若不使用观察者模式，常见做法是在状态变更的核心方法里顺序调用各个动作：

```
order.setStatus(newStatus);
smsService.send(order);        // 发短信
pointsService.grant(order);    // 发积分
// 将来还要发优惠券、推消息、记日志……继续往后堆
```

这种写法会带来如下设计问题：

- **违反开闭原则**：每新增一种后续动作（发优惠券、推送消息等），都要改动状态变更这个核心方法，改动面大、回归风险高。
- **职责耦合**：状态变更的发布方被迫感知并直接依赖短信、积分等一众下游服务，「发布状态变更」与「如何响应变更」紧紧捆在一起，发布方无法独立演进与测试。
- **扩展与维护成本高**：后续动作越多，核心方法越臃肿；任一动作的改动都可能波及发布方，动作之间也容易相互牵连。

观察者模式将「订单状态变更」抽象为一个**事件**，由**发布者**对外发布；各后续动作分别抽象为相互独立的**监听者**，各自订阅该事件并独立响应。发布方只管发布事件，对「有哪些监听者、各自如何响应」一无所知，从而把「状态变更的发布」与「对变更的各种响应」彻底解耦。

## 二、遵循的设计原则

- **开闭原则（OCP）**：新增一种对状态变更的响应，只需新增一个带 `@EventListener` 方法的 `@Component` 监听者即可自动接入，无需改动发布者 `OrderStatusEventPublisher` 与调用方 `OrderStatusController`。体现位置：`SmsNotifyListener`、`PointsRewardListener` 各自独立订阅同一事件。
- **单一职责原则（SRP）**：发布者只负责「发布事件」（`OrderStatusEventPublisher#publish`），每个监听者只负责一个关注点（`SmsNotifyListener` 只发短信、`PointsRewardListener` 只发积分），事件对象 `OrderStatusChangedEvent` 只承载变更快照，三类职责彼此分离。
- **依赖倒置原则（DIP）**：发布者仅依赖 Spring 的事件发布抽象 `ApplicationEventPublisher`，不依赖任何具体监听者；监听者也只依赖事件对象，不感知发布者。双方均面向抽象而非彼此的具体实现。
- **松耦合（Loose Coupling）**：发布者与监听者之间唯一的耦合点是不可变的事件对象 `OrderStatusChangedEvent`，二者互不持有引用、可各自独立增减与测试。

## 三、优点与适用及不适用场景

**优点：**

- 发布方与响应方彻底解耦，发布方无需知道由谁响应、如何响应。
- 响应方可独立增减（新增/移除监听者无需改动发布方），符合开闭原则，扩展成本低。
- 同一事件可被多个相互独立的监听者各自处理，各关注点内聚、互不牵连，便于单独测试与复用。

**适用场景：**

- 一个对象状态变化需要通知多个其它对象，且通知方与被通知方需要解耦（如状态变更通知、领域事件广播、UI 数据绑定）。
- 响应动作的集合会随业务动态增减、希望「即插即用」的场景。

**不适用场景：**

- 响应动作固定且极少变化、数量很少时，引入事件机制反而增加抽象与排查成本，直接顺序调用更直观。
- 各监听者之间存在严格的执行先后或事务一致性要求时，基于松耦合广播的观察者模式难以保证顺序与原子性，需改用编排式流程或显式事务控制。
- 对调用链路要求强可追踪、强同步返回的场景，事件「发布后由各监听者异步/解耦处理」会使调用链与异常传播变得不直观。

## 四、参与角色与对应类

| 标准角色 | 职责 | 本示例对应类 |
| --- | --- | --- |
| Subject / Publisher（主题 / 发布者） | 在状态变更时对外发布事件，不感知具体观察者 | `com.example.patterns.behavioral.observer.OrderStatusEventPublisher` |
| Observer（观察者抽象） | 订阅并响应事件的统一约定 | Spring 事件机制的 `@EventListener` 方法（框架提供的观察者抽象） |
| ConcreteObserver（具体观察者） | 实现对事件的具体响应逻辑 | `SmsNotifyListener`（发短信）、`PointsRewardListener`（发积分） |
| Event（事件 / 通知消息） | 承载状态变更快照，作为发布方与监听方唯一的耦合点 | `com.example.patterns.behavioral.observer.OrderStatusChangedEvent` |
| Client / Trigger（触发方） | 接收外部请求并触发发布者发布事件 | `com.example.patterns.behavioral.observer.controller.OrderStatusController` |

> 关于角色取舍的说明：经典观察者模式中，「主题」通常自行维护一个观察者列表（`attach/detach/notify`），并在 `notify` 时逐个回调。本示例借助 Spring 的事件机制（`ApplicationEventPublisher` 发布 + `@EventListener` 订阅）来承担「观察者注册与多播」的职责，使发布者无需手工维护观察者列表、监听者可由容器自动注册。此为对经典结构的有意调整，目的是契合 Spring 容器的自动装配，让监听者「即插即用」并保持发布方与监听方的彻底解耦。
