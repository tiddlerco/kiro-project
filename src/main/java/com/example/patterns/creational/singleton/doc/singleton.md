# 单例模式 Singleton

业务场景：进程内的共享资源管理。应用运行期需要一份全局唯一、随处可取的对象来承载共享状态——例如全局配置项（功能开关、阈值、文案模板）或本地缓存（热点字典、临时计算结果）。这类对象若被反复创建，不仅浪费资源，更会导致「各处读到的状态不一致」。

本示例同时给出两种实现方式以作对照：

- `GlobalConfigManager`：依托 Spring 容器默认的单例作用域（singleton scope）实现，是单例模式在 Spring 生态中最自然的落地方式。
- `LocalCacheManager`：纯 Java 语言机制的经典手写单例，采用双重检查锁（DCL）保证线程安全。

## 一、解决的问题

某些对象在整个应用生命周期内只应存在一个实例，并需要被全局共享访问。若放任各处自由 `new`，会带来如下问题：

- **状态不一致**：配置、缓存等共享状态被分散到多个实例中，一处写入对另一处不可见，行为难以预测。
- **资源浪费**：重复创建本应唯一的重量级对象（如缓存容器、连接持有者），徒增内存与初始化开销。
- **缺乏统一访问入口**：调用方需自行管理对象的创建与传递，耦合度高、易出错。

单例模式通过「将实例创建的唯一入口收敛到类自身」来解决上述问题：

- 经典实现把构造方法私有化，仅经静态 `getInstance()` 暴露唯一实例（`LocalCacheManager`）；
- Spring 实现则把实例的创建与生命周期交由容器托管，容器的单例作用域天然保证「一个类型一个实例」（`GlobalConfigManager`），经依赖注入获取，无需手写单例控制代码。

## 二、遵循的设计原则

- **单一职责原则（SRP）**：单例类既负责保证「自身实例唯一」，又承载其领域职责（配置存取 / 缓存读写）。本示例中，实例唯一性的保证方式被清晰隔离——经典单例由私有构造方法 + `getInstance()` 的双重检查锁承担，Spring 单例则完全交由容器，业务方法（`getConfig`/`setConfig`、`put`/`get`）保持纯粹。
- **依赖倒置 / 控制反转（DIP / IoC）**：`GlobalConfigManager` 不自行管理生命周期，而是声明为 `@Component` 交由 Spring 容器创建与注入，调用方经 `@Resource` 面向对象注入获取，不感知实例化细节。体现位置：`SingletonDemoService` 经 `@Resource` 注入 `GlobalConfigManager` 与 `ApplicationContext`。
- **开闭原则（OCP）**：单例的获取入口稳定（容器注入 / `getInstance()`），新增配置项或缓存项只需调用既有读写方法，无需修改单例类的实例控制逻辑。

## 三、优点与适用及不适用场景

**优点：**

- 全应用唯一实例，保证共享状态一致，避免重复创建带来的资源浪费。
- 提供统一、稳定的全局访问入口，调用方无需关心实例的创建与传递。
- Spring 单例还可天然享受依赖注入、AOP 增强、生命周期回调等容器能力。

**适用场景：**

- 需要全局唯一且被广泛共享的对象，如全局配置、本地缓存、ID 生成器、线程池持有者等。
- 创建成本较高、希望全程复用同一实例的重量级对象。

**不适用场景：**

- 对象需要携带请求级 / 会话级的可变状态时，全局单例会引发并发污染——此类场景应使用原型作用域或方法局部对象。
- 需要多实例以支持隔离（如多租户各自一份配置）时，单例会成为瓶颈。
- 经典手写单例若对「反射破坏私有构造」「序列化生成新实例」敏感，DCL 写法无法天然抵御，应改用枚举单例。
- 滥用单例承载过多全局可变状态，会形成隐式的全局耦合，降低可测试性与可维护性。

## 四、参与角色与对应类

| 标准角色 | 职责 | 本示例对应类 |
| --- | --- | --- |
| Singleton（单例 · Spring 容器实现） | 由容器保证全应用唯一，承载全局配置存取 | `com.example.patterns.creational.singleton.GlobalConfigManager` |
| Singleton（单例 · 经典手写实现） | 以私有构造 + 双重检查锁保证全进程唯一，承载本地缓存读写 | `com.example.patterns.creational.singleton.LocalCacheManager` |
| Client / Service（演示服务） | 委派配置读写，并对比两种单例「多次获取是否引用相等」 | `com.example.patterns.creational.singleton.service.SingletonDemoService` |
| Result（对比结果） | 承载两种单例的引用相等结论及各自获取方式说明 | `com.example.patterns.creational.singleton.domain.SingletonComparisonResult` |

> 关于两种实现取舍的说明：经典单例（`LocalCacheManager`）以 DCL 完整展示了「可见性 + 禁止指令重排 + 双重判空」等线程安全关键考量，适合教学；而在 Spring 工程实践中，更推荐 `GlobalConfigManager` 这种「交由容器托管」的方式——它无需手写任何单例控制代码，且天然融入依赖注入与 AOP 体系。本示例并列二者，正是为了对照「语言层手写单例」与「容器层托管单例」在实现方式与获取入口上的差异。
