# 装饰器模式（Decorator）

## 一、解决的问题

在「通知发送」这类场景中，基础能力（把通知发出去）是稳定的，但围绕它的增强能力却在不断变化：有时需要给正文追加签名，有时需要对正文加密，有时需要记录发送日志，更多时候需要这些能力的**任意组合**。

如果用继承来应对，会陷入「子类爆炸」：`签名发送器`、`加密发送器`、`签名+加密发送器`、`签名+加密+日志发送器`…… 能力每增加一项，组合数量呈指数增长，且组合方式在编译期被写死，无法在运行期按需拼装。

装饰器模式通过「把增强能力拆成一个个可独立叠加的装饰器、在运行期层层包裹基础组件」来解决这一问题：调用方传入一份能力列表，即可在不修改任何既有类的前提下，动态组合出所需的增强链路。

## 二、遵循的设计原则

- **开闭原则（OCP）**：新增一种增强能力，只需新增一个具体装饰器类，无需改动基础组件 `BaseNotifySender` 或其他既有装饰器。
- **单一职责原则（SRP）**：每个装饰器只负责一项增强（签名 / 加密 / 日志），基础组件只负责基础发送，职责清晰互不耦合。
- **组合优于继承**：抽象装饰器 `NotifyDecorator` 通过**构造方法注入**持有被装饰的 `NotifySender`（组合），而非继承具体组件来扩展，从而支持任意子集、任意顺序的能力叠加（对应需求 8.3）。
- **依赖倒置原则（DIP）**：装饰器与调用方均依赖抽象 `NotifySender` 接口，而非具体实现类，使装饰器既能包裹基础组件，也能包裹其他装饰器。

## 三、优点、适用场景与不适用场景

### 优点

- 增强能力可在运行期按需、任意组合，避免子类爆炸。
- 增强逻辑与基础逻辑解耦，各装饰器可独立开发、测试与复用。
- 新增能力对既有代码零侵入，符合开闭原则。

### 适用场景

- 需要在不影响其他对象的前提下，为单个对象**动态、透明**地添加职责。
- 增强能力存在大量可自由组合的子集，用继承会导致类数量失控。
- 希望增强能力可灵活撤销或调整叠加顺序。

### 不适用场景

- 增强能力组合固定且数量很少时，引入装饰器反而增加理解成本，直接编码更简单。
- 装饰器层数过多时，调试链路较长，问题定位需要逐层排查。
- 对装饰顺序高度敏感、且各层之间存在隐式依赖时，需谨慎设计以免叠加顺序引发难以察觉的副作用。

## 四、参与角色与对应类

| 角色 | 说明 | 对应类 |
| --- | --- | --- |
| 抽象组件（Component） | 抽象出「发送通知」的统一能力 | `NotifySender` |
| 具体组件（ConcreteComponent） | 装饰链最内层的真实基础发送逻辑 | `BaseNotifySender` |
| 抽象装饰器（Decorator） | 实现组件接口并以组合方式持有被装饰对象 | `NotifyDecorator` |
| 具体装饰器（ConcreteDecorator） | 在委派前后叠加签名能力 | `SignatureDecorator` |
| 具体装饰器（ConcreteDecorator） | 在委派前后叠加加密能力 | `EncryptDecorator` |
| 具体装饰器（ConcreteDecorator） | 在委派前后叠加日志能力 | `LogDecorator` |
| 装配中心 | 按能力列表以基础组件为内核逐层组装装饰链 | `DecoratorNotifyService` |
| 演示入口 | 暴露 HTTP 接口触发演示 | `DecoratorController` |

### 演示入口

- `POST /pattern/decorator/send`：请求体携带 `receiver`、`title`、`body` 与 `capabilities`（能力列表，如 `["SIGNATURE","ENCRYPT","LOG"]`），服务层据此组装装饰链并发送，返回包含最终正文与已应用能力的发送结果。
