# 模板方法模式（Template Method）

> 业务场景：企业多渠道支付对账。对账整体流程稳定不变（拉取对账文件 → 解析为标准流水 → 与平台流水逐笔比对 → 生成差异报告），而「文件如何拉取」「文件如何解析」因渠道（支付宝、微信等）格式各异。

## 一、解决的问题

不同支付渠道的对账文件格式互不相同：支付宝以英文逗号分隔、中文表头，微信以竖线分隔、英文表头。但对账的业务编排是一致的——都要经历「拉取、解析、比对、生成报告」四步。

若为每个渠道各写一套完整对账流程，则：

- 「比对、生成报告」等通用逻辑在各渠道间重复，难以维护；
- 各渠道流程步骤顺序无统一约束，容易被改乱，导致对账口径不一致。

模板方法模式把**稳定的算法骨架**固定在抽象类的模板方法中，把**易变的步骤**抽象出来交由子类实现，从而复用骨架、隔离差异、约束流程顺序。

## 二、遵循的设计原则

- **好莱坞原则（Don't call us, we'll call you）**：由父类模板方法反向调用子类实现的抽象步骤，控制流牢牢掌握在骨架手中。
- **开闭原则（OCP）**：新增渠道只需新增一个具体模板子类并实现抽象步骤，无需修改既有骨架与其他渠道。
- **单一职责原则（SRP）**：抽象类负责流程编排与通用步骤，具体类只负责本渠道差异步骤（拉取、解析）。
- **依赖倒置原则（DIP）**：上层路由服务依赖抽象模板 `AbstractReconcileTemplate`，而非具体渠道实现。

## 三、优点与适用及不适用场景

**优点**

- 一次性复用稳定骨架，消除多渠道流程间的重复代码；
- 通过 `final` 模板方法锁定步骤顺序，保证各渠道对账口径一致；
- 扩展新渠道成本低，符合开闭原则；
- 借助钩子方法（如 `loadPlatformRecords()`）提供可选扩展点，按需重写。

**适用场景**

- 多个实现共享同一套固定流程，仅个别步骤存在差异；
- 需要约束子类不得改变算法的整体结构与步骤顺序；
- 存在可选的扩展点，允许子类按需介入（钩子方法）。

**不适用场景**

- 流程步骤本身高度可变、无稳定骨架可抽象；
- 步骤差异更适合在运行期动态切换（此时策略模式更合适）；
- 继承层级过深或步骤过多，导致抽象类膨胀、子类理解成本上升。

## 四、参与角色与对应类

| 模式角色 | 对应类 | 职责 |
| --- | --- | --- |
| 抽象类（AbstractClass） | `AbstractReconcileTemplate` | 定义 `final reconcile()` 模板方法固定骨架，声明 `fetch()`/`parse()` 抽象步骤，提供 `compare()`/`generateReport()` 通用步骤与 `loadPlatformRecords()` 钩子方法 |
| 具体类（ConcreteClass） | `AlipayReconcileTemplate` | 实现支付宝渠道的文件拉取与逗号格式解析 |
| 具体类（ConcreteClass） | `WechatReconcileTemplate` | 实现微信渠道的文件拉取与竖线格式解析 |
| 路由服务 | `ReconcileService` | 聚合全部模板，按渠道标识建立路由表（重复渠道启动期快速失败），按渠道触发对账 |
| 演示控制器 | `ReconcileController` | 提供 `POST /pattern/template/reconcile` 演示入口，仅做路由分发 |
| 请求对象 | `ReconcileRequest` | 承载渠道标识，配合 `@Validated` 声明式校验 |
| 报告/记录/差异 | `ReconcileReport`、`ReconcileRecord`、`ReconcileDifference` | 承载对账过程与结论数据 |

### 演示方式

HTTP 触发：`POST /pattern/template/reconcile`，请求体 `{"channel": "alipay"}` 或 `{"channel": "wechat"}`，返回对应渠道的对账差异报告。未知渠道将返回业务异常提示。
